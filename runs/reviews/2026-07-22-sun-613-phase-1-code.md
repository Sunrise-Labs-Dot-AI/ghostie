# Adversarial code review, SUN-613 phase 1 (device identity)

PR: https://github.com/Sunrise-Labs-Dot-AI/ghostie/pull/15
Author lane: Claude (Opus 4.8). Second lane: Codex, independent of the author.
Date: 2026-07-22. **Verdict: BLOCK**, 3 CRITICAL, 6 WARNING. NOT MERGED.

## The two findings that falsify claims I made

**Finding 1: non-extractable does not mean unusable.** I wrote, in the code comments and the PR
body, that the persisted blob is "useless to any process that reads the file". That is false. The
blob is an encrypted capability that the SAME Secure Enclave can reconstruct: any same-user process
that reads it can call the initializer and request signatures. It is useless on ANOTHER machine,
which is a real and worthwhile property, but it is not the property I claimed. The default CryptoKit
initializer also applies no access control, so there is no user-presence requirement at signing time.

**Finding 2: `MESSAGES_FOR_AI_HOME` redirects the production lookup path.** I named a test
`testNoEnvironmentVariableCanSubstituteAKey`. An environment variable can in fact redirect the blob
path to attacker-controlled bytes, and `loadKey()` pins no expected public-key fingerprint, so a
same-user process can substitute a valid key of its own and the swap is silent. The test name
asserts something the code does not do.

Both are my errors, not reviewer overreach. A claim in a comment is a claim.

## Assessment

Finding 4 is the one I had already spotted while waiting for the review: `loadKey()` returns nil for
every failure mode except non-empty-but-corrupt, and nil means mint. So "a corrupt blob never
re-mints" holds for exactly one of the six corruption cases.

Finding 3 is worth reading carefully rather than dismissing as a nit. "Flag off means no key
material" is currently true only because nothing calls the production path at all. That is a
property of dead code, not of a gate. It would stop being true the moment phase 2 wires it up,
which is precisely when nobody would re-check it.

Findings 1, 2, and 8 together say the same thing from three angles: this is a send-authority key
with no binding between the identity and the thing that is supposed to hold it. Fixing that is an
access-control policy decision (user presence at signing? on which devices?) plus fingerprint
pinning plus a key generation counter, and it is a design question, not a patch.

Finding 7 is a documentation-consistency failure that would have shipped a spec contradicting its
own implementation.

## Disposition

Not merged. The branch is pushed at `f434fd2` with CI green, so nothing is stranded. The fixes
needed are substantial enough that they are their own piece of work rather than a round of patches
on this diff.
