# Plan review, SUN-613 phase 1 read-only (redaction projection)

Author lane: Claude (Opus 4.8). Second lane: Codex, independent. 2026-07-22.
**Verdict: BLOCK**, 7 CRITICAL, 5 WARNING. Plan rewritten; phase boundary moved.

## What the review changed (not just the field list)

Findings 7, 8, 9 together showed the phase itself was mis-scoped: writing per-draft snapshot FILES,
wired into DraftStore, before the phase-2 data plane (push vs pull) and queue schema (manifest,
generation, tombstones) are decided, is premature AND persists message bodies to disk for no runtime
consumer, which is the same dead-code problem the plan claimed to avoid. The reviewer's explicit
carve-out: if redaction must land first, limit phase 1 to a PURE projection with adversarial tests,
no wiring, no bodies on disk. That is what shipped.

So phase 1 became the transport-independent privacy core: the redaction projection and its tests.
Everything with a runtime footprint (publisher, storage, manifest, credential, flag, wiring) moved
to phase 2 as one vertical slice with its consumer.

## What landed

`menubar/Sources/MessagesForAIMenu/RelaySnapshot.swift`: `RelaySnapshot`, `RelayRecipient`,
`RelayContextMessage`, `RelayLifecycle`, and the pure `project(from:originDeviceID:)`. No file I/O,
no identity read, no flag, no wiring.

`menubar/Tests/MessagesForAIMenuTests/RelaySnapshotTests.swift`: 14 tests. The load-bearing ones
assert the EXACT encoded key set of each projection type (so a future field cannot leak by default),
that no sensitive value from a fully-loaded hostile draft appears anywhere in the JSON, that an
unnamed group becomes a count, that an unknown context sender becomes a pseudonym, that the
projection creates no file, and that script/bidi-shaped text is carried inert.

## Phase-2 requirements carried forward

Recorded in `docs/plans/cross-device-draft-sync-spec.md` and the phase-1 plan: data-plane decision
before storage (7); per-origin manifest with generation/publish-time/membership/tombstones (6, 8);
serial publisher actor, generation-ordered, Sendable projections (5); flag-off purge on transition
and startup, reads gated independently of deletion (4); availability flag AND local opt-in before any
copy (3); publish only pending/scheduled/held or a short TTL (11); draft-id validation, platform
namespacing, O_NOFOLLOW, temp cleanup (10).

# Code review of the implementation (same day)

Second lane reviewed the diff and returned BLOCK, 2 CRITICAL, 7 WARNING, 1 NOTE. The important
finding was conceptual, not a bug.

**Finding 2 (CRITICAL), accepted as a correction to the claim.** An allowlist of field NAMES does
not bound field VALUES. A draft body is free text and can contain anyone's phone number, and showing
the body is the whole point of a draft-review feature, so "no third-party identity crosses, no matter
what the draft contains" is impossible. The contract is now stated honestly in two tiers: structural
identifiers (paths, thread ids, guids, message ids, reaction authors, approval tags, execution
fields) are excluded by construction and asserted by a recursive string walk; message text and the
recipient label are content, carried verbatim, and may contain PII the user themselves wrote. A test
pins the passthrough so the scope cannot drift.

**Finding 1 (CRITICAL), real leak, fixed.** A group draft whose `imessage_group` struct was dropped
but whose raw `to_handle` still carries the group binding would have leaked its guid or participant
handles through the direct-recipient path. Now detected with the app's own `isIMessageGroupDraft`
(which covers exactly this degenerate case), and such a draft projects to a generic "Group thread"
label.

Fixed WARNINGs: handle-shaped `sender_name` and group participant "names" are pseudonymised or
counted rather than printed (finding 2 structural sub-cases); the digest is now length-prefixed at
every level with distinct nil markers and includes `schema_version`, so `"Bob:hello"` and
`"Bob","hello"` cannot collide (finding 6); lifecycle reuses `Draft.isSent`/`isHeld`/`isScheduled`
with explicit precedence and drops the never-emitted `.failed` (finding 9); `RelayQuotedPreview` is
now implemented rather than only documented (finding 8); bidi controls are stripped from short
identity labels while message bodies are left for the phase-3 renderer to bidi-isolate (finding 7).
Tests rewritten to decode and walk every string scalar rather than substring-match escaped JSON
(finding 5), to check the key set with the optional both present and absent (finding 3), and to lock
value types via a golden-shape test (finding 4). The purity test is now honestly labelled as
necessary-not-sufficient (finding 10).

Verification: 750 Swift tests, 0 failures; 18 in `RelaySnapshotTests`.
