Author lane: Claude (Opus 4.8). Second lane: Codex (`codex exec`, read-only), independent of the author.
Date: 2026-07-22. **Verdict: BLOCK**, 10 CRITICAL, 4 WARNING.

## What the measurement showed

The design decision was settled by probing rather than by argument:

```
SecureEnclave.isAvailable = true
SE P256: created=yes sign/verify=true blobBytes=284
SE P256: restored-from-blob verify=true, pubkey stable across restore=true
legacy keychain add OSStatus = 0
legacy readback: synchronizable=nil   <- attribute not recorded
legacy readback: accessible=nil       <- attribute not recorded
data-protection keychain add OSStatus = -34018 (errSecMissingEntitlement)
```

So the Keychain options were: one that needs an entitlement we refuse to ship, and one that cannot
verify the property it claims. The enclave needs neither and gives a stronger property.

## Full second-lane output

## Finding 1: The macOS backup guarantee is false as written

**Severity:** CRITICAL

**Issue:** `kSecAttrSynchronizable: false` does prevent a newly added item from synchronizing and excludes synchronized items from a query. It does not make `kSecAttrAccessibleAfterFirstUnlockThisDeviceOnly` effective on macOS. Apple states that `kSecAttrAccessible` applies on macOS only when `kSecUseDataProtectionKeychain: true` or synchronization is true. This plan sets synchronization false and omits the data-protection keychain selector. [Apple documentation](https://developer.apple.com/documentation/security/ksecattraccessible), [data-protection keychain documentation](https://developer.apple.com/documentation/security/ksecusedataprotectionkeychain)

**Failure scenario:** The generic-password item lands in the legacy file-based login keychain. The plan believes it is nonmigratory, but the `ThisDeviceOnly` class was not applied. A restored keychain makes the raw signing key available on another Mac, which can sign as the original device.

**Suggested fix:** Put `kSecUseDataProtectionKeychain: true` on every add, read, update, and delete operation. Keep synchronization explicitly false. Use `WhenUnlockedThisDeviceOnly` unless background signing is actually required. Verify the returned live attributes on a signed build.

## Finding 2: Keychain authorization is unspecified

**Severity:** CRITICAL

**Issue:** The plan names neither an access group nor an access-control policy. Ghostie deliberately gives every inner Mach-O the same code-signing identifier. Once the data-protection keychain is selected, the default access group may therefore expose the key to more bundled executables than intended. The legacy-keychain alternative has different ACL behavior, which is also unspecified. [Apple access-group documentation](https://developer.apple.com/documentation/security/ksecattraccessgroup)

**Failure scenario:** A remotely reachable or Claude-driven MCP binary shares the effective keychain access group, reads the raw Ed25519 private key, and signs approvals without the menu-bar approval UI.

**Suggested fix:** Define the precise authorized executable set. Give only the menu-bar binary a dedicated keychain-access-group entitlement, and test that every bundled MCP and daemon is denied. If the entitlement model cannot provide that separation without a provisioning-profile change, stop and redesign before storing send authority there.

## Finding 3: The Secure Enclave alternative was dismissed without analysis

**Severity:** WARNING

**Issue:** `Curve25519.Signing.PrivateKey` requires storing extractable raw key material. Apple’s Secure Enclave supports P-256 signing, not Ed25519. The plan never explains why extractable Ed25519 is preferable for two Apple-silicon Macs and a WebCrypto client. It also chooses `AfterFirstUnlock`, leaving the key readable while the screen is locked, although pairing and approval are interactive operations. [Apple Secure Enclave documentation](https://developer.apple.com/documentation/security/protecting-keys-with-the-secure-enclave), [CryptoKit P-256 interface](https://developer.apple.com/documentation/cryptokit/secureenclave/p256/signing)

**Failure scenario:** Code execution inside an authorized process after the first unlock extracts the raw Ed25519 key and retains indefinite approval authority.

**Suggested fix:** Evaluate `SecureEnclave.P256.Signing.PrivateKey` with user-presence access control and WebCrypto P-256 interoperability. If Ed25519 remains, document the threat-model tradeoff and use the most restrictive accessibility class compatible with the actual workflow.

## Finding 4: The Keychain test proves only that the test helper returns what it was told

**Severity:** WARNING

**Issue:** Asserting on `RelayKeychain.addQuery()` is self-fulfilling. It does not prove which macOS keychain implementation was selected, which access group owns the item, whether the accessibility class was honored, or whether some other call path omitted an attribute. A live integration test that may skip on every CI machine is not an acceptance gate.

**Failure scenario:** The dictionary test stays green while the actual item is placed in the legacy keychain or becomes accessible to an unintended executable. Release validation never detects it because the only live test skipped.

**Suggested fix:** Retain the dictionary unit test, but add a required signed-host integration gate. Add a uniquely named item, retrieve it with `kSecReturnAttributes`, enumerate with `kSecAttrSynchronizableAny`, verify synchronization, accessibility, data-protection keychain behavior, access group, key length, and public-key correspondence, then delete the test item.

## Finding 5: Pre-existing synchronized items are ignored, not handled

**Severity:** WARNING

**Issue:** Synchronizability is part of a generic-password item’s composite primary key, so synchronized and local-only records with the same service/account can coexist. A false query correctly ignores the synchronized record, but the plan never enumerates it or defines remediation. [Apple generic-password documentation](https://developer.apple.com/documentation/security/ksecclassgenericpassword), [synchronization documentation](https://developer.apple.com/documentation/security/ksecattrsynchronizable)

**Failure scenario:** A previous experimental build left a synchronized key with the same service and account. Enrollment creates a second local key, while manual validation or later maintenance code finds the wrong record. A future query regression silently revives the synchronized credential.

**Suggested fix:** During enrollment, enumerate matching records using `kSecAttrSynchronizableAny`. Refuse enrollment with explicit remediation if any synchronized collision exists. Do not automatically delete it, because deletion of synchronized items propagates to other devices.

## Finding 6: The test environment variable is a production key-injection backdoor

**Severity:** CRITICAL

**Issue:** `MFA_TEST_RELAY_DEVICE_KEY` lets process environment select a deterministic private key. Mirroring an existing unsafe seam does not make another credential override acceptable.

**Failure scenario:** A same-user process arranges for Ghostie to launch with an attacker-known key. James completes pairing against that apparently normal Ghostie instance. The attacker now possesses the enrolled private key and can forge approvals indefinitely.

**Suggested fix:** Use dependency injection from the XCTest target. Do not consult an environment variable in release code. At minimum, compile the override only into debug/test builds and make release builds fail if the variable is present.

## Finding 7: Six uncommitted digits are breakable by an active transport attacker

**Severity:** CRITICAL

**Issue:** The advertised one-in-a-million security applies only when the attacker must commit before learning the honest contributions. This plan has no commitment exchange. A man in the middle can generate attacker keys `X` and `Y` until `SAS(A,X) == SAS(B,Y)`. Because the output is only about 20 bits, two lists of roughly 1,000 candidate keys already have a substantial chance of an intersection. Domain separation does nothing against this attack. The plan also defines no attempt limit.

**Failure scenario:** The attacker delays pairing briefly, substitutes `X` toward Mac A and `Y` toward Mac B, finds a matching six-digit collision, and both screens display the same code. Each Mac then enrolls a key controlled by the attacker.

**Suggested fix:** Use a standard committed SAS protocol. Both sides must commit to fresh random contributions and the full enrollment transcript before either reveals. Add one-shot sessions, expiration, mismatch abort, and rate limits. Alternatively compare a high-entropy fingerprint through a genuinely out-of-band QR or equivalent channel.

## Finding 8: The SAS is not bound to a session or enrollment record

**Severity:** CRITICAL

**Issue:** Sorting public keys leaks nothing because they are public, but it deliberately discards direction and role. The hash omits device IDs, roles, protocol versions, capabilities, session nonces, and a pairing identifier. It is static for the lifetime of two keys. It also proves no possession of either private key.

**Failure scenario:** A transport attacker leaves the two displayed keys unchanged while altering `device_id`, `role`, or `relay_protocol` in the stored enrollment record. The SAS still matches. Parallel or replayed ceremonies using the same keys also produce indistinguishable codes.

**Suggested fix:** Define a canonical transcript containing a fresh pairing ID, both nonces, role-labeled device IDs, public keys, protocol ranges, and capabilities. Both parties must sign challenges and the final transcript. Derive the SAS from that transcript, reject self-pairing and duplicate keys, and activate enrollment only after both proof-of-possession signatures and human confirmation succeed.

## Finding 9: There is no implementable pairing ceremony in this phase

**Severity:** CRITICAL

**Issue:** The build list contains key storage, a pure SAS function, and a JSON store. It contains no enrollment coordinator, UI, public-key exchange mechanism, pending state, confirmation action, or recovery path. The phase simultaneously forbids networking and claims two Macs will prove identity to each other.

**Failure scenario:** Implementation finishes exactly as planned, but there is no production path by which the M1 learns the M4 public key or either screen reaches the claimed six-digit comparison. The stated production validation cannot be performed.

**Suggested fix:** Specify the complete exchange mechanism and state machine: start, exchange commitments, reveal transcript, verify proof of possession, display SAS, confirm on both devices, atomically activate, cancel, expire, and resume or discard partial pairing. If key exchange is deferred, rename this phase to identity primitives and remove every claim that pairing is delivered.

## Finding 10: The protocol-version check does not detect old executors

**Severity:** CRITICAL

**Issue:** A self-reported integer is negotiation metadata, not attestation. Even when signed, it proves only what the enrolling app claims. It says nothing about stale menu-bar processes, MCP binaries, daemons, or Claude configurations launching old paths. This is precisely the “old executor never announces” problem the canonical spec identified. The plan merely asks a newer process to announce on its behalf. It also accepts arbitrary future versions because it checks only the minimum.

**Failure scenario:** The current menu-bar app truthfully announces version 1 while an old iMessage MCP remains configured. A draft lands in the spoke’s ordinary drafts directory, the old MCP ignores `relay_executor` and sends it before phase 2 moves it, then the hub sends it again.

**Suggested fix:** Stop describing the field as trustworthy. Bind the negotiated range into the signed pairing transcript, reject incompatible future versions, and make rollout safety independent of announcements. Require an enforced local audit of every send-capable binary, configured path, and running process, or design the spoke path so relay-destined drafts never enter any directory old executors scan. A pre-existing setting understood by old executors, such as mandatory approval, may be an additional belt, not the primary guarantee.

## Finding 11: `devices.json` destroys the existing local-forgery boundary

**Severity:** CRITICAL

**Issue:** Mode 0600, ownership checks, and `O_NOFOLLOW` do not stop another process running as the same user from replacing a regular file. The existing approval design explicitly treats same-user JSON writes as hostile and relies on a Keychain secret. Making an owner-writable JSON file the public-key trust root throws that protection away.

**Failure scenario:** A local process replaces an enrolled peer’s public key with its own while preserving owner, mode, schema, device ID, and `revoked_at: null`. It signs an approval with the matching private key. The hub verifies against the forged trust record and mints the real local approval HMAC.

**Suggested fix:** Authenticate the trust store with a least-privilege Keychain key, or store each peer trust anchor directly in the Keychain. Bind every authorization-relevant field into the authenticated value. File hardening remains necessary for availability, but it is not integrity protection.

## Finding 12: The mutable-store protocol is missing

**Severity:** WARNING

**Issue:** Phase 0’s immutable create-and-link discipline cannot simply be reused for a mutable enrollment and revocation database. The plan specifies neither interprocess locking, atomic replacement, durability, conflict detection, duplicate handling, nor the read-to-use boundary. It also duplicates role state between settings and peer records without declaring which source wins.

**Failure scenario:** Enrollment and revocation concurrently read the same file. Enrollment writes last and loses the revocation. Alternatively, duplicate records exist for one device, one active and one revoked, and verifier behavior depends on array order. A crash between truncate and write destroys the trust database.

**Suggested fix:** Add a store generation and an interprocess lock. Update through an `openat`-scoped temporary file, complete-write loop, `fsync`, atomic rename, and parent-directory `fsync`. Reject duplicate device IDs, duplicate keys, malformed base64, non-32-byte keys, invalid timestamps, unknown roles, and inconsistent `hub_device_id` relationships. Recheck the active record generation immediately before minting local approval authority.

## Finding 13: Device ID and signing-key continuity are undefined

**Severity:** WARNING

**Issue:** `device.json` can survive a machine restore while a correctly configured `ThisDeviceOnly` private key cannot. The plan does not define whether a missing key under an existing device ID is rotation, corruption, replacement, or a new device. Pairing is also written independently on each machine, so partial success creates asymmetric trust.

**Failure scenario:** A restored Mac retains its old device ID, generates a fresh key during enrollment, and is now represented by the same ID with different keys on different peers. Revocation and signature lookup become order-dependent, and existing pairings fail unpredictably.

**Suggested fix:** Introduce a `key_id` or public-key fingerprint, key generation, and pairing ID. Bind them to the device ID. Never silently mint a replacement key under an established identity. Missing keys require explicit recovery and re-pairing, or a new device ID. Represent enrollment as pending until both sides have confirmed the identical transcript.

## Finding 14: `revoked_at` is not a revocation design

**Severity:** CRITICAL

**Issue:** A nullable timestamp in a replaceable JSON file is neither monotonic nor rollback-resistant. Missing, null, malformed, duplicate, and conflicting records have no specified semantics. Restoring an older valid trust-store snapshot clears revocation even if the file is authenticated.

**Failure scenario:** An enrolled iPhone is revoked. An old backup of `devices.json`, where `revoked_at` was null, is restored. The old key becomes active and its signatures are accepted again.

**Suggested fix:** Model explicit `pending`, `active`, and `revoked` states with pairing ID, key generation, and monotonic trust-store generation. Keep revocation tombstones. Anchor the latest accepted generation or revoked key IDs in the Keychain so an older authenticated snapshot cannot resurrect authority.

## Finding 15: Phase 3 cannot safely use the phase 1 model

**Severity:** CRITICAL

**Issue:** The future approval envelope omits the hub/audience ID, pairing ID, key ID or generation, executor ID, and assignment revision. A nonce is meaningless without an atomic persistent replay ledger, yet the canonical delivery plan postpones ledger persistence and revocation to phase 4 after signed approvals appear in phase 3. A 120-second TTL bounds replay time; it does not prevent replay.

**Failure scenario:** An approval captured for hub A is replayed to hub B, where the same approver key is enrolled and the same draft ID and digest exist. After a restart, an in-memory nonce set is empty, so the captured approval is accepted again. An approval for assignment revision 1 can also authorize revision 2.

**Suggested fix:** Move persistent replay protection and active-key revocation checks into the same phase as the first remote approval. Canonically sign the action, protocol version, hub ID, approver device ID, pairing ID, key generation, draft ID, payload digest, executor ID, assignment revision, issued-at, expiry, and a 128-bit nonce. Atomically consume the nonce before minting the local tag, then recheck the active trust generation immediately before send.
